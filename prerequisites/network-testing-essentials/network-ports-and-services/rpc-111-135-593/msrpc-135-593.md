# MSRPC (135, 593)



**Microsoft Remote Procedure Call (MSRPC)** is an Application Layer protocol that lets a program on one machine call a function/procedure that runs on another machine as if it were a local call. The calling program doesn't know the call crossed a network — the RPC runtime handles all the transport underneath.

It is Microsoft's implementation of the Open Group DCE/RPC standardOkay. .&#x20;

You'll encounter it on essentially every Windows machine — it is the backbone of almost every Windows service: Active Directory, SMB, WMI, DCOM, Task Scheduler, Print Spooler, SAMR, LSASS, SCM, and more all communicate over RPC.

{% hint style="info" %}
_Port 135 is the doorway. Even if all other ports are filtered, 135 being open tells you the target is Windows, and from 135 you can enumerate every registered RPC service on that machine._
{% endhint %}

## Key Terminology

### Marshalling and Unmarshalling

**Marshalling** → Converting a function call into data for network transmission.

**Unmarshalling** → Converts data back to a function call for execution.&#x20;

```
CreateUser("John","Password123")
          ↓
      Marshalling
          ↓
Operation: CreateUser
User: John
Pass: Password123
          ↓
       Network
          ↓
    Unmarshalling
          ↓
CreateUser("John","Password123")
          ↓
      Execution
```

### Client and Server&#x20;

In MSRPC there are always two sides.

**RPC Client** ⇒ Requests a function to be executed on the remote machine.&#x20;

**RPC Server** ⇒ Provides the function result after execution.&#x20;

### RPC Interface

An RPC Interface is a collection of functions.

E.g.,&#x20;

{% code overflow="wrap" %}
```
Printer Interface

AddPrinter()
DeletePrinter()
ListPrinters()
```
{% endcode %}

### Interface UUID

Every RPC interface has a unique identifier called UUID.&#x20;

{% hint style="info" %}
_No two interfaces should have the same UUID._&#x20;
{% endhint %}

An Endpoint is simply the location where an RPC interface can be reached.

**E.g. - 1** ,&#x20;

**E.g., - 2**,

### Endpoint Mapper (135)

The Endpoint Mapper is the **"phone book"** of MSRPC. It's an RPC service itself, always running on **TCP port 135**, and it maps interface UUIDs to their actual network endpoints (port numbers, named pipes).

     │   │ {367abb81-...}    │  2.0     │ TCP:49667        │    │

### Binding&#x20;

Before a client can call a function, it must connect to the interface and this process is called binding.&#x20;

### RPC Runtime

The **heart** of MSRPC on Windows. Every process that uses RPC loads `rpcrt4.dll`:

**its job is to:**&#x20;

* Manages connections and binding handles
* Performs endpoint resolution (talks to the Endpoint Mapper)
* Negotiates authentication (NTLM, Kerberos, etc.)
* Manages thread pools for concurrent requests

> _Think of it as a dispatcher._&#x20;

### Transport Deep Dive

#### ncacn\_ip\_tcp (Direct TCP)

```
Client ──► Port 135 (EPM) ──► "Where is SAMR?" ──► "Port 49667"
Client ──► Port 49667 ──► RPC calls over TCP
```

* **Most direct** — client talks straight to the RPC server
* Requires port 135 open + dynamic port range open
* Used by: `impacket-rpcdump`, `impacket-samrdump`, most Impacket tools

#### ncacn\_np (Named Pipes over SMB)

```
Client ──► Port 445 (SMB) ──► Authenticates to SMB ──► Opens \\target\IPC$
       ──► Opens named pipe (e.g., \pipe\samr)
       ──► RPC calls flow through the pipe
```

* RPC rides on top of SMB — **inherits SMB authentication**
* Only needs port 445 open (no dynamic ports needed!)
* Used by: `rpcclient`, PsExec, smbexec, many Windows admin tools
* Named pipes have their own **ACLs** — access can be restricted per-pipe

#### ncacn\_http (RPC over HTTP)

```
Client ──► Port 593 (RPC-HTTP EPM) or 443 ──► HTTP CONNECT/RPC_IN_DATA
       ──► RPC calls tunneled through HTTP
```

* Designed to traverse firewalls and proxies
* Used by: Exchange/Outlook Anywhere, Outlook MAPI/HTTP
* Less common in pentesting, but important for Exchange attacks

The Endpoint Mapper itself is an RPC interface with UUID: `e1af8308-5d1f-11c9-91a4-08002b14a0fa`

### Stubs (Client & Server)

```
CLIENT SIDE:                           SERVER SIDE:
┌────────────────────┐                 ┌────────────────────┐
│   Client Stub      │                 │   Server Stub      │
│                    │                 │                    │
│ 1. Takes function  │                 │ 1. Receives NDR    │
│    parameters      │                 │    byte stream     │
│ 2. Marshals into   │    Network      │ 2. Unmarshals into │
│    NDR format      │ ──────────────► │    native params   │
│ 3. Sends via       │                 │ 3. Calls real      │
│    transport       │                 │    server function │
│                    │                 │ 4. Marshals result │
│ 4. Receives        │ ◄────────────── │ 5. Sends response  │
│    response        │                 │                    │
│ 5. Unmarshals      │                 │                    │
│    result          │                 │                    │
└────────────────────┘                 └────────────────────│
```

### Interface Definition language (IDL)

IDL is the **contract** between client and server. It defines:

* What functions are available
* What parameters they take
* What data types are used
* The **UUID** (unique identifier) for the interface

```c
/* Example: A simplified IDL definition */
[
    uuid(12345678-1234-ABCD-EF00-0123456789AB),
    version(1.0)
]
interface SampleInterface
{
    HRESULT GetUserInfo([in] long userId, [out] wchar_t* userName);
    HRESULT CreateUser([in] wchar_t* userName, [in] wchar_t* password);
}
```

Microsoft's **MIDL compiler** takes IDL files and generates **client stubs** and **server stubs** — the glue code that handles marshaling.

{% hint style="info" %}
_Each function in an RPC interface is identified by an **operation number (opnum)**. When you see `opnum 15` in a packet capture, that's the 16th function (0-indexed) defined in the interface's IDL. Tools like Wireshark decode opnums to human-readable function names for well-known interfaces._
{% endhint %}

### Authentication

RPC supports authentication.

A client can prove its identity.

Common Windows methods:

* NTLM
* Kerberos
* Negotiate (SPNEGO)

### Versioning

Interfaces have versions.

Example:

```
Printer Interface

Version 1.0
Version 2.0
Version 3.0
```

Client can ask:

```
Do you support v2.0?
```

This prevents compatibility issues.

## The complete flow&#x20;

{% code overflow="wrap" %}
```
 CLIENT                                                    SERVER
 ──────                                                    ──────
 1. Application calls AddUser("john", "pass123")
         │
 2. Client stub marshals params into NDR format
         │
 3. RPC Runtime queries Endpoint Mapper (port 135):
    "Where is interface {UUID}?"
         │
 4. EPM responds: "TCP port 49667"
         │
 5. RPC Runtime connects to port 49667
         │
 6. Authentication negotiation (NTLM/Kerberos)
         │
 7. RPC Request PDU sent ──────────────────────────►  8.  RPC Runtime receives PDU
    (contains: interface UUID,                             │
     opnum, NDR-encoded params)                       9.  Server stub unmarshals params
                                                           │
                                                      10. Server executes 
                                                      AddUser("john","pass123")
                                                           │
                                                      11. Server stub marshals return value
                                                           │
 13. Client stub unmarshals   ◄────────────────────── 12. RPC Response PDU sent
     return value
         │
 14. Application gets result
```
{% endcode %}

### Working

{% code overflow="wrap" %}
```
CLIENT                                               TARGET SERVER
──────                                               ─────────────
1. Connect to TCP 445 (SMB)
2. SMB authentication (NTLM/Kerberos/Null)
3. Connect to IPC$ share:
   NET USE \\target\IPC$ "" /u:""     (null session)
   NET USE \\target\IPC$ "pass" /u:"user"  (authenticated)
4. Open named pipe:
   \\target\pipe\samr
5. RPC calls flow through the pipe
```
{% endcode %}

### Named Pipe ACLs

Each named pipe has a **DACL** (Discretionary Access Control List) that controls who can connect:

* `Everyone` / `Anonymous Logon` — null session access (if permitted)
* `Authenticated Users` — any domain/local authenticated user
* `Administrators` — admin-only access
* **Default on modern Windows**: Most pipes require authentication; anonymous access is heavily restricted

## Authentication in MSRPC

### Security Support Providers (SSPs)

MSRPC uses **SSPs** (via the SSPI interface) for authentication. These are the same authentication packages used by all of Windows:

| SSP                    | Auth Mechanism                           | RPC Constant                    | When Used                                  |
| ---------------------- | ---------------------------------------- | ------------------------------- | ------------------------------------------ |
| **NTLM**               | Challenge-response hash                  | `RPC_C_AUTHN_WINNT` (10)        | Legacy environments, workgroups, fallback  |
| **Kerberos**           | Ticket-based (TGT/TGS)                   | `RPC_C_AUTHN_GSS_KERBEROS` (16) | Domain environments, preferred             |
| **Negotiate (SPNEGO)** | Tries Kerberos first, falls back to NTLM | `RPC_C_AUTHN_GSS_NEGOTIATE` (9) | Default — most common in practice          |
| **Schannel**           | TLS/SSL certificate-based                | `RPC_C_AUTHN_GSS_SCHANNEL` (14) | Machine certificates, LDAPS-like scenarios |
| **None**               | No authentication                        | `RPC_C_AUTHN_NONE` (0)          | Anonymous / null session access            |

### Authentication Levels&#x20;

Beyond _who_ you authenticate as, MSRPC defines _how much_ protection the connection gets:

| Level | Name            | Protection | Description                                 |
| ----- | --------------- | ---------- | ------------------------------------------- |
| 1     | `NONE`          | ❌          | No authentication at all                    |
| 2     | `CONNECT`       | 🔑         | Authenticates once at connection time only  |
| 3     | `CALL`          | 🔑         | Authenticates at the start of each RPC call |
| 4     | `PKT`           | 🔑📦       | Authenticates every packet                  |
| 5     | `PKT_INTEGRITY` | 🔑📦✅      | Auth + integrity (detects tampering)        |
| 6     | `PKT_PRIVACY`   | 🔑📦✅🔒    | Auth + integrity + encryption               |

{% hint style="info" %}
* _**Level 2 (CONNECT)** is the most common default — authentication happens once, then the data flows unprotected. This means RPC traffic can potentially be **sniffed** or **tampered** with._
* _**Level 6 (PKT\_PRIVACY)** is used for sensitive operations (like DRSUAPI/DCSync) and encrypts everything._
* _When you perform NTLM relay attacks against RPC interfaces, the auth level matters — you can only relay to services that accept the same or lower auth level._
{% endhint %}
